# Current-state revalidation map

This ledger translates the 15 numbered findings from `docs/14-protocol-audit-report.md` into present-tree checkpoints that later slices can replay.
It is written for later audit authors who need to decide whether a historical issue is already re-proven in tracked evidence, still only partially retired, or still needs attached-hardware confirmation.
After reading it, a later slice author should be able to pick a finding number, rerun the named check, and know which current file owns the truth today.

## Reader contract

- `docs/14-protocol-audit-report.md` is historical input, not current truth.
- `docs/15-milestone-10-protocol-remediation.md` is useful historical context, but a checked batch alone does not retire a finding unless the current tree still exposes matching source or test evidence.
- Every row below names the current owner files plus a fresh re-check command and evidence anchor.
- When the current tree does not carry enough fresh evidence to retire a finding honestly, the row stays `not re-proven` or `hardware needed` instead of inheriting a historical “fixed” claim.

## Approved verdict buckets

| Current verdict bucket | Use when | Closure rule |
| --- | --- | --- |
| `re-proven` | The current tree has tracked source and/or named local regression evidence that directly retires the historical finding without needing a live-device qualifier. | A later slice may treat the historical finding as retired if the named command still passes and the cited owner files still match the claim. |
| `partially re-proven` | The current tree clearly improved or structurally fixed the issue, but the exact historical failure mode still lacks one direct replay, fault injection, or live capture. | Keep the row open until the missing targeted replay is added or rerun; do not collapse this into a full pass. |
| `not re-proven` | The current tree may have related remediation notes, but M001 does not yet carry tracked evidence strong enough to say the original finding is retired. | Later slices must gather a targeted local regression or source-backed proof before treating the issue as closed. |
| `hardware needed` | Honest closure requires an attached device, live HID/browser exchange, or another physical/runtime capture that M001 does not include. | Keep release wording conservative until the named live probe is rerun against hardware. |
| `claim drift closed` | The historical finding was documentation or version-identity drift and the current claim surfaces now agree with the shipped tree. | Re-open only if the cited docs/metadata/version files drift again. |
